## 1. Общее описание системы и функционал

Приложение **ReportSystem** — это кроссплатформенная информационная система (десктопное приложение), разработанная на платформе **.NET 8** с использованием фреймворка **Avalonia UI** для автоматизации процессов регистрации, мониторинга и обработки обращений о нарушениях дисциплины (буллинг, списывание, прогулы и т.д.) в образовательной организации.

### Ролевая модель и разграничение прав доступа
Система поддерживает три роли пользователей, каждая из которых имеет свой уникальный интерфейс и доступный функционал:
1. **Ученик (Student):**
   - Просмотр личного дашборда (статистика только по собственным обращениям).
   - Создание и отправка обращений (с возможностью сделать обращение **анонимным**).
   - Просмотр истории собственных поданных обращений и их текущих статусов.
2. **Преподаватель (Teacher):**
   - Просмотр общего дашборда со статистикой нарушений по всей школе/колледжу.
   - Просмотр всех поданных обращений (включая анонимные, но для анонимных скрыты данные автора).
3. **Администратор (Admin):**
   - Полная статистика по системе на главном дашборде.
   - Просмотр и фильтрация всех обращений в системе.
   - Управление статусом обращений (смена статуса: `New` → `InProgress` → `Resolved` / `Rejected`).
   - Управление пользователями: просмотр списка учетных записей, изменение роли пользователей (`Student`, `Teacher`, `Admin`).

## 2. Архитектура приложения: паттерн MVVM в Avalonia UI

В основе клиентской части приложения лежит паттерн **MVVM (Model-View-ViewModel)**. Это стандарт проектирования для современных UI-платформ, который разделяет графический интерфейс, логику представления и данные.

```
┌─────────────────────────────────────────────────────────────┐
│                       ПРЕДСТАВЛЕНИЕ (View)                  │
│  - MainWindow.axaml, LoginWindow.axaml, страницы .axaml     │
│  - Описывает внешний вид UI с помощью разметки XAML        │
└──────────────────────────────┬──────────────────────────────┘
                               │ (Связывание данных / Binding)
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                 МОДЕЛЬ ПРЕДСТАВЛЕНИЯ (ViewModel)            │
│  - MainWindowViewModel.cs, LoginViewModel.cs и т.д.          │
│  - Хранит состояние интерфейса и обрабатывает команды (Commands)│
└──────────────────────────────┬──────────────────────────────┘
                               │ (Вызовы методов / EF Core)
                               ▼
┌─────────────────────────────────────────────────────────────┐
│                         МОДЕЛЬ (Model)                      │
│  - User.cs, Report.cs, Category.cs (Сущности бд)             │
│  - Описывает бизнес-логику и структуры данных               │
└─────────────────────────────────────────────────────────────┘
```

### За счет чего работает связывание (Binding) и реактивность?
Для реализации MVVM используется библиотека **CommunityToolkit.Mvvm**:
- Все **ViewModel** наследуются от класса `ObservableObject` (или `ViewModelBase`, который сам наследуется от `ObservableObject`).
- Изменение свойств на форме происходит за счет метода `SetProperty(ref _fieldName, value)`. Этот метод автоматически генерирует событие `PropertyChanged`. Уведомленный интерфейс (XAML) мгновенно перерисовывает измененное значение.
- Действия пользователя (нажатия кнопок) привязываются через интерфейс `IRelayCommand` (класс `RelayCommand` или `RelayCommand<T>`), что избавляет нас от необходимости писать обработчики кликов в файлах логики страниц (Code-Behind).

### Как работает динамическая навигация?
Переключение страниц в главном меню реализовано с помощью компонента `ContentControl` в файле [MainWindow.axaml](file:///c:/Projects/ReportsSystem/Views/MainWindow.axaml):
1. `ContentControl` связан с помощью Binding со свойством `CurrentPage` типа `object?` во ViewModel.
2. При смене выбранного пункта меню во ViewModel свойству `CurrentPage` присваивается новая ViewModel нужной страницы (например, `CreateReportViewModel`).
3. Специальный класс [ViewLocator.cs](file:///c:/Projects/ReportsSystem/ViewLocator.cs) перехватывает изменение значения и сопоставляет класс ViewModel с соответствующей View (XAML-файлом страницы) с помощью конструкции `switch`.
4. Новая страница динамически загружается в основную область окна, при этом ее `DataContext` автоматически заполняется этой ViewModel.

## 3. Устройство Базы Данных (Схема и Связи)

Для хранения данных используется реляционная СУБД **PostgreSQL**, развернутая на облачном хостинге **Neon.tech**.
Взаимодействие с базой данных происходит через ORM **Entity Framework Core 8** с использованием Npgsql-провайдера.

### Описание сущностей (Таблиц)

1. **Users (Пользователи):**
   - `Id` (int, PK) — суррогатный первичный ключ.
   - `FullName` (string) — ФИО пользователя.
   - `Login` (string, Unique) — уникальный логин для авторизации.
   - `PasswordHash` (string) — хэш пароля (получен алгоритмом BCrypt).
   - `RoleId` (int, FK) — ссылка на таблицу ролей.
   - `Email` (string, Nullable) — электронная почта.
   - `CreatedAt` (DateTime) — дата и время регистрации аккаунта.

2. **Roles (Роли):**
   - `Id` (int, PK) — идентификатор роли.
   - `Name` (string) — название роли (`Admin`, `Teacher`, `Student`).

3. **Categories (Категории нарушений):**
   - `Id` (int, PK) — идентификатор категории.
   - `Name` (string) — наименование категории (например, «Буллинг», «Списывание»).
   - `Description` (string) — описание категории.
   - `SeverityLevel` (int) — уровень важности/серьезности (1 - низкий, 2 - средний, 3 - высокий).

4. **ReportStatuses (Статусы обращений):**
   - `Id` (int, PK) — идентификатор статуса.
   - `Name` (string) — название статуса (`New`, `InProgress`, `Resolved`, `Rejected`).

5. **Reports (Обращения о нарушениях):**
   - `Id` (int, PK) — идентификатор обращения.
   - `AuthorId` (int, FK) — ссылка на пользователя (`Users`), создавшего обращение.
   - `CategoryId` (int, FK) — ссылка на категорию нарушения (`Categories`).
   - `Description` (string) — детальное описание ситуации.
   - `StatusId` (int, FK) — ссылка на статус обращения (`ReportStatuses`).
   - `IsAnonymous` (bool) — флаг анонимности (если `true`, имя автора не отображается преподавателям и администраторам в списках).
   - `CreatedAt` (DateTime) — дата подачи обращения.
   - `ReviewedById` (int, FK, Nullable) — ссылка на администратора/преподавателя (`Users`), который проверил обращение.
   - `ResolvedAt` (DateTime, Nullable) — дата решения проблемы.
   - `ModeratorComment` (string, Nullable) — комментарий проверяющего.

6. **Evidences (Доказательства нарушений):**
   - `Id` (int, PK) — идентификатор доказательства.
   - `ReportId` (int, FK) — ссылка на обращение (`Reports`), к которому прикреплен файл.
   - `FilePath` (string) — путь к загруженному файлу.
   - `FileType` (string) — тип файла (расширение).
   - `UploadedAt` (DateTime) — дата загрузки.

### Настройка связей (Fluent API) в `ApplicationDbContext`

Когда сущность имеет несколько связей с одной и той же таблицей, встроенные механизмы автоматического сопоставления EF Core не справляются. В нашем случае класс `Report` имеет две связи с таблицей `Users`:
* `Author` (кто написал) — связь один-ко-многим.
* `ReviewedBy` (кто проверил) — связь один-ко-многим.

Для явного указания этих связей используется **Fluent API** в переопределенном методе `OnModelCreating` файла [ApplicationDbContext.cs](file:///c:/Projects/ReportsSystem/Data/ApplicationDbContext.cs):

```csharp
modelBuilder.Entity<Report>()
    .HasOne(r => r.Author)
    .WithMany(u => u.Reports)
    .HasForeignKey(r => r.AuthorId);

modelBuilder.Entity<Report>()
    .HasOne(r => r.ReviewedBy)
    .WithMany(u => u.ReviewedReports)
    .HasForeignKey(r => r.ReviewedById);
```

### Автоматическая инициализация (Data Seeding)
В приложении настроена автоматическая генерация базовых данных в СУБД прямо при применении миграций. В методе `OnModelCreating` вызываются методы `HasData` для заполнения:
* **Ролей:** Admin (1), Teacher (2), Student (3).
* **Статусов:** New (1), InProgress (2), Resolved (3), Rejected (4).
* **Категорий:** Списывание, Буллинг, Прогулы, Хулиганство, Другое.

## 4. Подробный разбор ключевых процессов (Data Flow)

### 🔑 Процесс 1: Запуск приложения и Авторизация
1. При запуске программы метод `OnFrameworkInitializationCompleted` в [App.axaml.cs](file:///c:/Projects/ReportsSystem/App.axaml.cs) назначает стартовым окном `LoginWindow`.
2. Пользователь может ввести данные или нажать «Зарегистрироваться», что скроет окно входа и откроет `RegisterWindow`.
3. **Регистрация:** При вводе данных в [RegisterViewModel.cs](file:///c:/Projects/ReportsSystem/ViewModels/RegisterViewModel.cs), пароль хэшируется библиотекой **BCrypt.Net-Next** с помощью соли (`BC.HashPassword(Password)`). Пользователь сохраняется в БД с ролью `Student` (по умолчанию).
4. **Вход:** В [LoginViewModel.cs](file:///c:/Projects/ReportsSystem/ViewModels/LoginViewModel.cs) выполняется запрос к базе данных: `db.Users.Include(u => u.Role).FirstOrDefault(u => u.Login == Login)`.
5. С помощью `BCrypt.Verify(Password, user.PasswordHash)` проверяется пароль. Если хэши совпадают, генерируется событие `LoginSucceeded`.
6. Класс `LoginWindow.axaml.cs` подписывается на это событие, закрывает окно входа и открывает `MainWindow(user)`, передавая объект вошедшего пользователя в конструктор.

---

### 📝 Процесс 2: Подача обращения (Роль Ученика)
1. Ученик переходит на страницу «Создать жалобу» (`CreateReportViewModel`).
2. При открытии страницы из базы данных подгружается список доступных категорий нарушений (`db.Categories.ToList()`).
3. Пользователь выбирает категорию из выпадающего списка `ComboBox`, пишет текст описания и опционально включает галочку «Подать анонимно» (`IsAnonymous = true`).
4. При нажатии «Отправить» вызывается команда `SubmitReport`:
   - Находится статус `New` в таблице `ReportStatuses`.
   - Создается новый экземпляр класса `Report`. В качестве автора указывается `_currentUser.Id`.
   - Запись добавляется в контекст базы данных: `db.Reports.Add(report)`.
   - Вызывается `db.SaveChanges()` для отправки данных в СУБД PostgreSQL.
   - Пользователю выводится сообщение об успешной отправке, а поля ввода очищаются.

---

### 🛡️ Процесс 3: Проверка и Модерация (Роль Администратора)
1. Администратор видит все обращения в системе через страницу `AdminReportsPage` (`AdminReportsViewModel`).
2. При загрузке страницы выполняется запрос с фильтрацией:
   ```csharp
   var query = db.Reports.Include(r => r.Status).AsEnumerable();
   ```
   - Если введен текст в поисковую строку, данные фильтруются по содержанию описания нарушения.
   - Если выбран фильтр по статусу (например, «InProgress»), отбираются только обращения с этим статусом.
3. При выборе конкретной строки в таблице `DataGrid` администратор может нажать кнопку для смены статуса (например, «В работу» или «Решено»). При этом:
   - В БД обновляется поле `StatusId` обращения.
   - Если выбран статус `Resolved` (Решено) или `Rejected` (Отклонено), автоматически проставляется дата закрытия обращения: `report.ResolvedAt = DateTime.UtcNow`.
   - Метод `db.SaveChanges()` фиксирует изменения в СУБД, а список на экране обновляется.

## 6. Нюансы реализации («Подводные камни» проекта)

При защите проекта будьте готовы к тому, что преподаватель может заметить расхождения между проектными планами и реальным кодом. Вот ключевые моменты, которые нужно знать:

1. **Таблица `Evidences` (Доказательства):**
   * *Как обстоят дела в коде:* Модель `Evidence` создана в базе данных, и в `Report` есть свойство `List<Evidence> Evidences`. Однако в графическом интерфейсе (UI) страницы создания жалобы и модерации функционал прикрепления файлов (Drag & Drop, выбор картинок) в данный момент **отсутствует**.
   * *Как отвечать на защите:* «В БД спроектирована и подготовлена структура для хранения файлов доказательств (таблица `Evidences`). На данном этапе UI-слой для прикрепления файлов находится в разработке и запланирован на следующий релиз. В данный момент пользователи отправляют только текстовые описания».

2. **Загрузка данных в память (`AsEnumerable()`):**
   * *Как обстоят дела в коде:* При поиске и фильтрации обращений/пользователей во ViewModel-ях вызывается метод `.AsEnumerable()`, после чего выполняется фильтрация (`query.Where(...)`). Это означает, что фильтрация происходит не на стороне базы данных PostgreSQL (SQL-запросом), а в оперативной памяти приложения.
   * *Как отвечать на защите:* «Для обеспечения корректной работы регистронезависимого поиска на русском языке с использованием встроенных функций C# (.Contains с параметром `OrdinalIgnoreCase`), коллекция выгружается в память через `.AsEnumerable()`. В будущем для высоконагруженной системы этот процесс будет оптимизирован с помощью переноса поиска на сторону СУБД (например, через ILIKE или полнотекстовый поиск PostgreSQL)».

## 🎓 Шпаргалка для защиты (Ответы на возможные вопросы комиссии)

### Вопрос 1: Что такое паттерн MVVM и почему вы выбрали именно его?
> **Ответ:** MVVM (Model-View-ViewModel) — это шаблон архитектуры, разделяющий визуальную часть (View), бизнес-логику (Model) и прослойку взаимодействия между ними (ViewModel). Мы выбрали его, потому что он является стандартным и рекомендуемым для Avalonia UI. Он позволяет отделить дизайн от логики, делает код чистым, тестируемым и поддерживаемым.

### Вопрос 2: Как данные из текстового поля в XAML попадают в переменную в C#?
> **Ответ:** За счет механизма **двусторонней привязки данных (Two-Way Binding)**. В XAML-коде у элемента `TextBox` прописано свойство `Text="{Binding Description}"`. Когда пользователь вводит текст на форме, Avalonia автоматически обновляет значение свойства `Description` во ViewModel. И наоборот.

### Вопрос 3: Зачем нужно наследоваться от `ObservableObject` во ViewModel?
> **Ответ:** Чтобы работал механизм оповещения UI об изменении свойств. Класс `ObservableObject` (из библиотеки CommunityToolkit.Mvvm) реализует стандартный интерфейс `INotifyPropertyChanged`. Когда мы меняем свойство через метод `SetProperty`, класс генерирует событие, сообщающее графической оболочке, что данные обновились и их нужно перерисовать на экране.

### Вопрос 4: Как работает авторизация и защита паролей в вашем приложении?
> **Ответ:** Пароли пользователей никогда не хранятся в базе данных в открытом виде. При регистрации мы берем введенный пароль и хэшируем его с помощью криптостойкого алгоритма **BCrypt** (`BCrypt.HashPassword`). В базу данных записывается только хэш. При входе мы берем введенный пароль, хэшируем его с той же солью и сверяем с хэшем из базы через метод `BCrypt.Verify`. Если они совпадают — авторизация успешна.

### Вопрос 5: Что такое ORM и зачем нужен Entity Framework Core?
> **Ответ:** ORM (Object-Relational Mapping) — объектно-реляционное отображение. Оно позволяет работать с базой данных как с набором обычных C#-классов (коллекций объектов), избавляя разработчика от необходимости писать SQL-запросы вручную. Entity Framework Core сам генерирует SQL-запросы для вставки, обновления, удаления и выборки данных на основе нашего C#-кода.

### Вопрос 6: Зачем в `ApplicationDbContext` переопределен метод `OnModelCreating`?
> **Ответ:** В этом методе с помощью **Fluent API** настраиваются специфические правила для БД, которые EF Core не может определить автоматически. Например, у нас настроена связь «один-ко-многим» для двух полей обращения (`AuthorId` и `ReviewedById`), ссылающихся на одну и ту же таблицу пользователей `Users`. Также здесь настроено автоматическое первичное наполнение (seeding) таблиц ролей, категорий и статусов.

### Вопрос 7: Как реализовано разграничение прав доступа для ролей Ученик, Преподаватель и Администратор?
> **Ответ:** Разграничение настроено на уровне отображения бокового меню в [MainWindow.axaml](file:///c:/Projects/ReportsSystem/Views/MainWindow.axaml). С помощью свойства `IsVisible` контейнеров `StackPanel` и привязок к свойствам `IsStudent`, `IsTeacher`, `IsAdmin` во ViewModel, каждому пользователю отображаются только те кнопки меню, которые соответствуют его роли. При нажатии кнопок навигация (`NavigateTo`) также жестко проверяет роль пользователя перед созданием ViewModel конкретной страницы.

### Вопрос 8: Где хранится база данных и как приложение с ней связывается?
> **Ответ:** База данных PostgreSQL развернута на облачной платформе Neon.tech. Приложение подключается к ней через интернет по строке подключения (Connection String), указанной в методе `OnConfiguring` класса `ApplicationDbContext`.

### Вопрос 9: Что такое миграции в Entity Framework Core?
> **Ответ:** Миграции — это инструмент EF Core, который позволяет отслеживать изменения в структуре классов (моделей) и автоматически переносить эти изменения в схему таблиц базы данных. Каждая миграция генерирует C#-код, описывающий, какие таблицы нужно создать, изменить или удалить.

### Вопрос 10: Как работает анонимность при подаче жалобы?
> **Ответ:** При создании жалобы ученик может поставить флаг `IsAnonymous = true`. В базе данных сохраняется реальный `AuthorId` (чтобы ученик мог видеть жалобу в разделе «Мои обращения»), но на страницах для преподавателя и администратора при отображении списка жалоб имя автора скрывается или заменяется на слово «Анонимно» на уровне графического интерфейса.

## 8. Инструкция по развертыванию и запуску

Для запуска проекта на локальном компьютере разработчика или экзаменатора выполните следующие шаги:

1. **Установите требования:**
   - [.NET 8.0 SDK](https://dotnet.microsoft.com/download/dotnet/8.0)
   - pgAdmin или СУБД PostgreSQL (если планируется локальная база данных). В противном случае используется облачная СУБД.

2. **Восстановите NuGet-пакеты:**
   ```bash
   dotnet restore
   ```

3. **Сформируйте базу данных:**
   Если вы хотите применить текущую схему БД к новой или существующей базе данных, выполните:
   ```bash
   dotnet ef database update
   ```
   *Примечание:* Миграции содержат автозаполнение (data seeding), поэтому таблицы ролей, категорий и статусов заполнятся автоматически.

4. **Запустите приложение:**
   ```bash
   dotnet run
   ```
